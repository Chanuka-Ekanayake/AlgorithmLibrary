# Two-Phase Commit (2PC) Complexity Analysis

## 1. Overview
The Two-Phase Commit protocol is evaluated differently from standard central algorithms. Instead of just algorithmic Time (Big-O) complexity based on input size, distributed algorithms are typically evaluated based on **Message Complexity** (the number of messages exchanged over the network), **Latency/Time Complexity** (the number of network round trips), and **Space Complexity** (the amount of durable storage required per transaction).

Let $N$ be the number of participants (Resource Managers) in the distributed transaction.

---

## 2. Message Complexity

Message complexity refers to the total number of network messages generated by a single execution of the 2PC protocol.

### Happy Path (All Commit)
1. **Phase 1 (Prepare):**
   - Coordinator sends `PREPARE` to $N$ participants: $N$ messages.
   - Participants send `VOTE_COMMIT` to Coordinator: $N$ messages.
2. **Phase 2 (Commit):**
   - Coordinator sends `GLOBAL_COMMIT` to $N$ participants: $N$ messages.
   - Participants send `ACK` to Coordinator: $N$ messages (Acknowledgments are required in robust implementations to allow the coordinator to garbage-collect the transaction log).

**Total Messages (Happy Path):** $N + N + N + N = 4N$ messages.
**Message Complexity:** $\mathcal{O}(N)$

### Optimization: Read-Only Transactions
If a participant evaluates a transaction and determines it is completely read-only (no state changes), it can vote `VOTE_READ_ONLY`. By doing so, it exempts itself from Phase 2. The coordinator does not need to send the global decision to that participant.
If $R$ participants vote read-only:
Total Messages: $(N) + (N) + (N-R) + (N-R) = 4N - 2R$.

### Failure Scenarios (Abort)
If a transaction aborts (e.g., one node sends `VOTE_ABORT`):
- `PREPARE` sent to $N$ nodes.
- At least 1 `VOTE_ABORT` and $N-1$ other votes (Commit/Abort/None).
- `GLOBAL_ABORT` sent to all $N$ nodes.
- `ACK` from all $N$ nodes.
Message count remains $\mathcal{O}(N)$ bounded by roughly $4N$.

---

## 3. Latency (Time) Complexity

Latency in distributed systems is often measured in Network Round Trip Times (RTT). An RTT is the time it takes for a message to go from node A to node B and for the response to return.

### Transaction Latency
1. **Phase 1 RTT:**
   - Message goes from Coordinator -> Participant (`PREPARE`).
   - Message returns from Participant -> Coordinator (`VOTE`).
   - **Cost: 1 RTT.**
2. **Phase 2 RTT:**
   - Message goes from Coordinator -> Participant (`GLOBAL_DECISION`).
   - Message returns from Participant -> Coordinator (`ACK`).
   - **Cost: 1 RTT.**

**Total Latency:** **2 RTTs.**
Wait time is driven by the slowest participant in the distributed system. Since the coordinator must wait for *all* participants to vote before proceeding to Phase 2, the actual time taken is:
$T_{prepare} + \max(T_{participant\_1}, T_{participant\_2}, ..., T_{participant\_N}) + T_{commit}$.

### Blocking Time
The crucial latency aspect of 2PC is the **Blocking Time**, which is the duration a participant holds locks on its local resources.
- A participant acquires locks when it receives the `PREPARE` message and logs its `VOTE_COMMIT`.
- It releases these locks only after it receives the `GLOBAL_COMMIT` or `GLOBAL_ABORT` message.
- Thus, the blocking window spans a full network round trip (from voting to receiving the decision), plus the time the Coordinator takes to write the decision to its disk. 

In a Wide Area Network (WAN), an RTT might be 100ms. If multiple transactions compete for the same rows, throughput will be severely throttled because locks are held for at least 100ms per transaction.

## 4. Space / Storage Complexity

To survive crashes, nodes must write the transaction state to non-volatile storage (a Write-Ahead Log or WAL).

### Coordinator Logging
- Must log `START_TRANSACTION` (optional but common).
- Must log the global decision: `GLOBAL_COMMIT` or `GLOBAL_ABORT`.
- Storage per transaction: $\mathcal{O}(1)$ (one or two log records per transaction).

### Participant Logging
- Must log the prepared state (`READY` record, containing lock metadata and undo/redo info).
- Must log the final state (`COMMITTED` or `ABORTED` record).
- Storage per transaction per participant: $\mathcal{O}(1)$ records, but the size depends on the transaction payload.

Overall system storage overhead is $\mathcal{O}(N)$ across all nodes per transaction, but $\mathcal{O}(1)$ per specific node.

## 5. Summary Table

| Metric | Complexity | Notes |
| :--- | :--- | :--- |
| **Message Complexity** | $\mathcal{O}(N)$ | $\approx 4N$ messages for $N$ participants. |
| **Network Latency** | $\mathcal{O}(1)$ RTTs | Exactly 2 RTTs. The absolute time depends on the slowest node. |
| **Disk I/O Costs** | $\mathcal{O}(1)$ fsyncs | Coordinator: 1 forced write. Participants: 2 forced writes (prepare, commit). |
| **Space Overhead** | $\mathcal{O}(1)$ per node | Minimal log entries per transaction. |
| **Availability (Fault Tolerance)** | Low | Susceptible to Coordinator failure ("Blocking problem"). |

## 6. Comparison with Other Consensus Protocols

To understand where 2PC falls short, it is useful to compare it with other protocols:

### Two-Phase Commit (2PC)
- **Goal:** Strict Atomicity / Consistency.
- **Problem:** Blocking. If the Coordinator dies during Phase 2, participants hold locks indefinitely. It is not fault-tolerant against coordinator crashes and network partitions.

### Three-Phase Commit (3PC)
- **Goal:** Non-blocking Atomicity.
- **Changes:** Adds an intermediate `PRE-COMMIT` phase to broadcast the intention to commit before actually committing.
- **Latency:** 3 RTTs instead of 2.
- **Message Complexity:** $\mathcal{O}(N)$ but higher constant factor.
- **Reality:** Rarely used because it is more complex, slower, and still fails under certain network partition scenarios.

### Sagas (Microservices Pattern)
- **Goal:** Eventual Consistency.
- **Changes:** Abandons distributed locks. Transactions are broken into local transactions. If a later step fails, compensating transactions are run to undo previous steps.
- **Latency:** Much lower, as no locks are held across network calls.
- **Reality:** Highly scalable, dominant pattern in modern microservice architectures, but sacrifices strict isolation.

### Paxos / Raft
- **Goal:** Fault-tolerant consensus on state/logs.
- **Changes:** Requires only a majority (quorum) of nodes to agree, rather than all nodes. 
- **Reality:** Used for distributed state machine replication (e.g., Zookeeper, etcd) rather than distributed database transaction atomicity. Some modern databases (like Spanner) combine **2PC for cross-shard atomicity** with **Paxos for shard replication fault-tolerance**.
