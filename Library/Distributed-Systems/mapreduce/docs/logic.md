# Algorithm Logic: MapReduce

MapReduce forces you to completely change how you think about data processing. Instead of telling the computer _how_ to iterate through the data, you define two specific behaviors: what to do with a single piece of data (**Map**), and what to do with a grouped set of results (**Reduce**).

In production systems (like Hadoop or Apache Spark), the MapReduce framework handles distribution, parallelization, and fault tolerance automatically; in this educational engine, we only simulate the distribution and parallelization logic and do not implement automatic retries or worker failure recovery.

## 1. The Map Phase (Extraction)

The `Map` function receives a single, raw input record (like a single line from a server access log) and extracts the relevant information. It outputs zero, one, or multiple **Key-Value pairs**.

**The Golden Rule of Mapping:** Mappers must be strictly stateless. A Mapper running on Server A cannot communicate with a Mapper running on Server B. They only know about the single record they are currently processing.

- **Input:** `10.0.0.1 - - [24/Feb/2026] "GET /api/models/v1 HTTP/1.1" 200`
- **Map Logic:** Extract the requested URL and emit a count of 1.
- **Output:** `("/api/models/v1", 1)`

---

## 2. The Shuffle Phase (The Framework's Magic)

Between the Map and Reduce phases lies the **Shuffle**. As a developer, you rarely write the code for this phase; the MapReduce engine (like Hadoop or Apache Spark) handles it.

The framework collects all the Key-Value pairs generated by thousands of Mappers across the cluster and groups them by their Key. It physically routes the data across the network so that all identical Keys end up on the exact same server.

- **Mapper 1 Output:** `("/api/models/v1", 1)`, `("/checkout", 1)`
- **Mapper 2 Output:** `("/api/models/v1", 1)`, `("/login", 1)`
- **Shuffle Output:**
  - `"/api/models/v1" -> [1, 1]`
  - `"/checkout" -> [1]`
  - `"/login" -> [1]`

---

## 3. The Reduce Phase (Aggregation)

The `Reduce` function receives a single Key and an _Iterable_ (list) of all the Values associated with that Key. Its job is to aggregate, summarize, or filter those values down to a final result.

Because the Shuffle phase guaranteed that _all_ data for a specific Key is present on this specific machine, the Reducer can confidently calculate the final total without needing to check with any other servers.

- **Input:** `"/api/models/v1", [1, 1]`
- **Reduce Logic:** Sum the array of values.
- **Output:** `("/api/models/v1", 2)`

---

## 4. Why Pure Functions Are Mandatory

In a distributed cluster, hardware fails constantly. If you have 1,000 servers, it is a statistical certainty that a hard drive will crash or a power supply will fail during a 10-hour processing job.

MapReduce survives this through **Deterministic Retries**.

If a Reducer function is halfway through processing a massive list of values and the server catches fire, the Orchestrator simply spins up a new server and hands it the exact same input data.

- If your Reducer relies on global variables, external databases, or random numbers (side-effects), the retry will produce a corrupted result.
- If your Reducer is a **pure function** (the same input _always_ produces the exact same output), the framework can retry the job infinitely without corrupting the final dataset.
