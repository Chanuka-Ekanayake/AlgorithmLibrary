# Three-Phase Commit (3PC) Complexity Analysis

## 1. Overview
The Three-Phase Commit protocol aims to eliminate the "blocking" state of its predecessor (2PC) while retaining strict distributed atomicity. To achieve this, it adds a third phase (Pre-Commit) and relies heavily on accurate timeouts. However, this theoretical safety comes at the cost of significantly increased Message Complexity and Time Complexity (Latency).

Let $N$ be the number of participants (Resource Managers) engaged in the distributed transaction.

---

## 2. Message Complexity

Message complexity refers to the total number of network messages generated by a single execution of the 3PC protocol over the network. 

### Happy Path (All nodes commit)
1. **Phase 1 (CanCommit):**
   - Coordinator sends `CAN_COMMIT` to $N$ participants: $N$ messages.
   - Participants evaluate and send `VOTE_YES` to Coordinator: $N$ messages.
2. **Phase 2 (PreCommit):**
   - Coordinator broadcasts `PRE_COMMIT` to $N$ participants: $N$ messages.
   - Participants log the state and send `ACK` to Coordinator: $N$ messages.
3. **Phase 3 (DoCommit):**
   - Coordinator broadcasts `DO_COMMIT` to $N$ participants: $N$ messages.
   - Participants commit and optionally send `ACK`: $N$ messages (required for log garbage collection).

**Total Messages (Happy Path):** $N + N + N + N + N + N = 6N$ messages.
**Message Complexity:** $\mathcal{O}(N)$

*Comparison:* 3PC generates roughly **$1.5\times$** the number of messages compared to 2PC ($6N$ vs $4N$).

### Failure Scenarios (Abort)
If a transaction aborts during Phase 1 (e.g., one participant votes `VOTE_NO` or times out):
- `CAN_COMMIT` sent to $N$ nodes.
- Coordinator receives at least 1 `VOTE_NO` and others.
- Coordinator broadcasts `ABORT` to $N$ nodes.
Total messages: $\approx 3N$, cutting the protocol short.

## 3. Latency (Time) Complexity

Latency is measured in Network Round Trip Times (RTT). In a wide-area network (WAN), a single RTT can be 50-150ms. RTT drastically bottleneck distributed databases because rows remain locked for the duration of the protocol.

### Transaction Latency
1. **Phase 1 RTT:** Coordinator -> Participant (`CAN_COMMIT`) and back (`VOTE`). **Cost: 1 RTT.**
2. **Phase 2 RTT:** Coordinator -> Participant (`PRE_COMMIT`) and back (`ACK`). **Cost: 1 RTT.**
3. **Phase 3 RTT:** Coordinator -> Participant (`DO_COMMIT`) and back (`ACK`). **Cost: 1 RTT.**

**Total Latency:** **3 RTTs.**

*Comparison:* This is a **$50\%$ increase in minimum latency** compared to 2PC (which uses 2 RTTs).

### The Blocking Trade-off
While Latency increases, the **Maximum Blocking Time** (in the event of a crash) theoretically drops.
- In 2PC, if the Coordinator crashes, nodes can block for an unbounded, infinite amount of time waiting for manual operator intervention.
- In 3PC, if the Coordinator crashes, recovery timeouts kick in. Because of the `PRECOMMIT` state, a participant or backup coordinator can deduce whether to abort or commit, thereby resolving the transaction locally after the timeout expires. The blocking time is bounded by the timeout configuration ($T_{max}$).

---

## 4. Space / Storage Complexity

Durability requires all state changes to be flushed to a non-volatile Write-Ahead Log (WAL). A "flush" or `fsync()` halts processing until the spinning disk or SSD acknowledges the write.

### Coordinator Logging
- Log `START_TRANSACTION`.
- Log `PRECOMMIT` decision.
- Log final `COMMIT` or `ABORT` decision.
**Cost:** $\mathcal{O}(1)$ records, but requires **2 synchronous fsyncs** per transaction (one before sending Pre-Commit, one before sending Do-Commit).

### Participant Logging
- Log `READY` state (including acquired locks and undo/redo vectors).
- Log `PRECOMMIT` state (recording the global intention).
- Log `COMMITTED` or `ABORTED` state.
**Cost:** $\mathcal{O}(1)$ records, but requires **3 synchronous fsyncs** per transaction.

*Comparison:* 3PC drastically increases disk I/O bottlenecks. While sequential SSD writes are fast, forcing an `fsync()` 3 separate times per transaction per node destroys throughput compared to modern batch-commit algorithms.

## 5. Summary Table

| Metric | 2PC (Two-Phase) | 3PC (Three-Phase) | Notes for 3PC |
| :--- | :--- | :--- | :--- |
| **Message Complexity** | $4N$ ($\mathcal{O}(N)$) | **$6N$ ($\mathcal{O}(N)$)** | 50% more network traffic. |
| **Network Latency** | 2 RTTs | **3 RTTs** | Drastically slower over WAN. |
| **Disk fsyncs (Participant)**| 2 sync writes | **3 sync writes** | High I/O penalty. |
| **Blocking Problem** | Unbounded | **Bounded** | Resolves Coordinator crashes automatically. |
| **Network Partitions**| Stalls safely | **Fails dangerously**| May result in data inconsistency (Split-Brain). |

## 6. The Fatal Flaw: Network Partitions

The theoretical elegance of 3PC is entirely destroyed by real-world networking realities, specifically **Network Partitions** (the 'P' in the CAP Theorem). 

3PC mathematically relies on the assumption that if Node A cannot reach Node B within a timeout, it strictly means Node B has crashed (Fail-Stop model). In reality, the network link could simply be severed, while both nodes are alive and processing.

**Scenario (Split-Brain):**
1. The Coordinator sends `PRE_COMMIT` to half the nodes (Partition A) before a network cable is cut. It fails to reach the other half (Partition B).
2. Partition A nodes receive `PRE_COMMIT`, transition to `PRECOMMIT`, and reply with `ACK`.
3. Partition B nodes time out waiting for `PRE_COMMIT`. Under 3PC rules, they safely decide to **ABORT**.
4. The Coordinator (in Partition A) receives `ACK`s from Partition A, assumes Partition B crashed, and sends `DO_COMMIT` to Partition A. Partition A nodes **COMMIT**.
5. **Result:** The database is permanently corrupted. Half the nodes committed, half aborted.

Because of this devastating vulnerability to network partitions, industry systems universally reject 3PC in favor of partition-tolerant consensus algorithms (like **Paxos** or **Raft**) combined with 2PC, or they abandon distributed ACID entirely in favor of **BASE** (Eventual Consistency) with patterns like **Sagas**.
